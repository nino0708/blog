# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Tokyo Towers Journal — a Japanese/English blog about Tokyo's major office towers and residential
skyscrapers. Static site (Astro) served from S3/CloudFront, with articles auto-generated by a
Lambda and CI/CD on AWS CodeBuild (not GitHub Actions). This repo (`nino0708/blog`) is the single
source for both the site and the generator/analytics Lambdas — CodeBuild builds directly from it.

Note: `README.md` describes the generator in Node (`index.mjs`/`prompts.mjs`) terms, but the
codebase has since been ported to Python (`generator/index.py`, `prompts.py`, `commons.py`) with
zero external dependencies (stdlib `urllib` for HTTP, `boto3` only for Secrets Manager/CodeBuild).
Trust the actual `.py` files over the README's Node references.

## Commands

```bash
cd site
npm install
npm run dev      # http://localhost:4321
npm run build    # astro build — also validates content collection schemas (site/src/content/config.ts)
npm run check    # astro check (type-checking .astro files)
npm run preview
```

Generate one article locally (no GitHub commit, writes directly into `site/src/content/`):

```bash
cd generator
ANTHROPIC_API_KEY=sk-ant-... python3 index.py --local
# ALLOW_UNVERIFIED=1 also allows verified:false buildings when running --local
```

There is no test suite or linter configured in this repo; `npm run build` (schema validation) and
`npm run check` are the closest correctness checks for the site.

## Architecture

Three independent pieces share this one repo:

1. **Content generation** (`generator/`, Python Lambda, runs daily 6:30 JST via EventBridge)
   - `index.py` — handler: picks the next un-published `verified:true` building from
     `generator/data/buildings.json`, asks Claude to write prose only (never invents facts/numbers),
     runs a second LLM fact-check pass in `prompts.py`, retries generation once on failure, and
     refuses to publish if the retry still fails. Repeats the same flow for the English article
     independently — Japanese can publish even if the English fact-check fails.
   - `commons.py` — fetches a CC-licensed hero image from Wikimedia Commons, avoiding images already
     used by existing articles.
   - `backfill_images.py` — run from `buildspec.yml` on every build (non-fatal) to self-heal articles
     missing a `heroImage`.
   - Commits generated Markdown straight to GitHub (`site/src/content/buildings/<slug>.md` and
     `buildings-en/<slug>.md`) via the REST API, then starts the CodeBuild project.

2. **Site** (`site/`, Astro)
   - `src/content/buildings/*.md` (JA, root path `/`) is the single source of truth for factual
     fields (completedYear, heightM, floorsAbove, etc.) — schema in `src/content/config.ts`.
   - `src/content/buildings-en/*.md` (EN, under `/en/`) holds **only** language-dependent text
     (title/summary/body/tags) and is matched to the JA entry by identical `slug`; numeric facts are
     never duplicated into English frontmatter — pages pull them from the JA collection at render time.
   - `src/i18n/ui.ts` centralizes UI strings, path translation, and spec formatting for both locales.
   - `expressways/`, `railways/`, `tourism/` collections exist for future content categories and share
     a generic schema (`genericCategorySchema` in `config.ts`).
   - SEO: canonical/OGP/Twitter card/JSON-LD per page, `sitemap.xml` with `hreflang` JA/EN/x-default.

3. **Analytics** (`analytics/index.py`, Python Lambda)
   - Aggregates CloudFront access logs from S3 (bot/asset traffic filtered out) into a weekly
     Markdown report committed to `reports/` in this repo — output feeds a separate downstream
     "business plan" routine that isn't part of this repo.

**Infra**: `infra/01-hosting.yaml` (S3 private via OAC + CloudFront + optional ACM/custom domain,
CloudFront Function rewrites `/path/` → `/path/index.html`) and `infra/02-pipeline.yaml`
(CodeBuild + generator Lambda + EventBridge schedule). `buildspec.yml` builds `site/`, syncs `dist/`
to S3 (long cache for hashed assets, short cache for HTML/XML/txt), and invalidates CloudFront.
CodeCommit is intentionally not used; GitHub + CodeBuild is the whole CI/CD path.

## Adding an article manually (see `templates/README.md` for full detail)

- Do **not** touch `site/src/` frontend code (layouts/components/i18n) for a normal article — the
  Astro content collection picks up new Markdown automatically.
- Pick an idea from `templates/article-ideas.json` (`status: "todo"`), copy
  `templates/building.ja.md` / `building.en.md` to `site/src/content/buildings(-en)/<slug>.md`
  with the same slug in both languages, fill in facts from real sources (delete any line you can't
  verify — never invent numbers), then `npm run dev`/`npm run build` to confirm before pushing.
- Manual articles don't need a `generator/data/buildings.json` entry; that seed file is only for
  buildings the daily auto-generation Lambda should pick up.

## Key invariants (do not break)

- Facts (year built, height, floors, area, etc.) come only from `generator/data/buildings.json` (auto
  path) or hand-verified frontmatter (manual path) — Claude/LLM output is prose only, never facts.
- Auto-publish requires `verified: true`; `verified: false` articles render an "unverified" badge and
  are excluded from the daily auto-publish selection.
- JA and EN articles for the same building must use the identical `slug`; English frontmatter must
  never carry numeric facts (they're sourced from the JA entry by slug at render time).
